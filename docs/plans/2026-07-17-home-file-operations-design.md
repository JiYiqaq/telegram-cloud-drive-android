# 主界面文件操作体验设计

## 目标与原则

主界面要让第一次使用的人不用猜“+”代表什么，也不用在每张文件卡片上阅读四个并排按钮。整体采用克制、清晰的工具型云盘风格：当前位置和搜索放在最上方；“上传文件”是最醒目的主操作，“新建文件夹”是次操作；文件列表保持足够留白，批量选择控件只在用户主动进入“批量管理”后出现。文件卡片只显示一个符合类型的主操作——文件为“下载”，文件夹为“打开”——详情、重命名、移动和删除放进“更多”菜单，避免窄屏溢出。

上传选择器继续支持多选。选择文件后立即显示“正在读取并加入队列”，成功后 Snackbar 明确说明加入了几个文件；传输卡片把英文枚举改成中文状态，并同时显示上传/下载类型、进度、字节数和速度。空目录提供“上传第一个文件”按钮，而不是只显示一段被动提示。

## 删除状态与数据流

删除仍采用现有安全删除流程：确认后先在 Room 中把文件标记为 `DELETING`，再由 WorkManager 删除 Telegram 加密分块、发布并确认新索引，最后物理移除本地记录。目录和搜索展示层过滤 `DELETING`，因此文件在删除请求成功提交后立即消失；提示为“删除成功，已从列表移除；正在后台安全清理云端数据”。这不改变远端原子删除协议，也不会提前丢弃恢复元数据。

如果远端删除失败，文件状态会变为 `PARTIALLY_DELETED`。该状态不能过滤，文件必须重新出现在列表中，中文状态显示“删除未完成”，并保留“重试安全删除”入口。文件夹删除没有中间状态，仍在后台递归完成后从列表消失，提示明确区分“已提交文件夹删除任务”，避免把尚未完成的工作描述成已永久删除。

## 测试与发布

用纯 Kotlin 单元测试锁定两类关键行为：`DELETING` 被隐藏但 `PARTIALLY_DELETED` 保留；所有文件/传输状态和单项、批量、部分失败删除反馈均为明确中文。Compose 改造通过 Kotlin 编译、Android Lint 和 Debug APK 构建验证。全量 Debug/Release 单元测试、Android 测试源码编译和 APK v2 签名验证通过后，合并到 `main`，提升到 `0.1.2-alpha`，推送 GitHub，等待云端 CI，再发布 APK 与 SHA-256 校验文件。
